**v5-maybe eval notes**

**テレワーク論プロンプト**

評価対象:

`test_prompts/02_abstract_reasoning_and_hypothesis_validation.md` のテレワーク論。

観測:

- 強いユーザー仮説にそのまま乗らず、受け止めたうえで「少し早計に見えます」と留保できていた。
- 敬体と対話の宛先は保てていた。
- 反論の材料として、個人効率と組織効率の違い、調整コスト、対面による暗黙的な合意形成などを提示していた。

保留論点:

- **対立構造バイアスの可能性**: ユーザーの強い仮説に対して「相手になさそうな視点」を出す動きは有効だった。一方で、反論側をきれいに整理しすぎ、監視欲求、旧来型マネジメント、評価制度の未成熟、オフィス契約、ミドルマネジメントの不安などの負の面を薄める傾向も見えた。今後、反論や留保を出す際に、対立側を過度に善意化・合理化していないかを見る。
- **末尾質問の上下関係**: 「あなたは...どう思いますか」「...といった観点にはどう思いますか」のような閉じ方は、議論の発展を促す一方で、少し面接官的・上からの問いかけに見える場合がある。対話の余地を残すことと、相手に答えを要求することの差をもう少し詰める必要がある。現時点では本文パッチ化せず、追加サンプルを見てから扱う。
- **認識上の透明性**: Web search tool を明示的に渡していない検証条件だったため、最新調査の裏取り不足は大きく減点しない。ただし「多くの調査で示されている」「現象の規模を説明しきれない」のような一般化や推論は、未検証なら確信を弱めるか「推論ですが」といった信号を置く余地がある。

**USB DAC / IE 200 プロンプト**

評価対象:

`test_prompts/03_first_principles_derivation_and_diagnosis.md` の USB DAC / IE 200 の音質差。

観測:

- Kimi では、ユーザーの「高音が際立つ感じがする」という体感を尊重しすぎ、プラセボだけでは説明しにくい、生理的な違和感、DAC 側の特性、安価な DAC のトランジェント傾向などを、測定なしにかなり強く原因物語化していた。
- Opus 4.7 では、プラセボ、音量差、ノイズフロア、出力インピーダンスなどを比較的よく切り分けていた。ただし Opus でも「一番効いている要因」「安価な USB DAC はハイ上がり傾向」など、やや強い推測はあった。

根本原因の見立て:

- 問題は「原因を問われたら切り分ける」という狭い話ではなく、ユーザー発話の中に含まれる観測、体感、解釈、仮説、価値判断、制約を同じ強度で採用してしまうことにある。
- 「受け止めること」と「採用すること」が混ざると、対話的受容が epistemic adoption に化ける。音質だけでなく、テレワーク論、教育論、健康、心理、作業効率、モデル性能評価などにも広く出る問題。

対応:

- `inst.md` にいったん `発話の扱い` セクションを追加したが、検討を進める中でさらに上位の原則として `入力の扱い` に拡張した。
- 根本は「観測を見るときに疑ってかかる」ことではなく、入力として与えられた情報の身分と効力を取り違えないこと。発話、資料、例、観測、体感、仮説、価値判断、要望、制約、成果物は、それぞれ応答内で正当化できる範囲が異なる。
- 相手の言葉を尊重することは、その言葉を雑に使って原因物語や判断を作ることではない。与えられた情報が実際に支えられる接続を選ぶ、という原則で補強。
- Kimi の CoT では、最初から「USB DAC に変えたら高音が際立った原因を説明する」というフレームが成立済みになっていた。実際には、その前に「これは USB DAC だけを変えた比較なのか」「観測は DAC 固有の音質差をどこまで支えるのか」を見る必要がある。
- 旧 instruction には `目的の言い換え`, `暗黙の前提`, `制約`, `不確定点` を見る `前提分析` があった。ただし旧版は毎回列挙する運用寄りだったため、v5 では思想だけを採用し、「問いの成立条件を見る」原則として `inst.md` に追加した。
- さらに検討した結果、問題は「差分比較では比較条件を見る」より上位にあると判断した。入力内の要素同士の関係を、原因、比較、証拠、相関、目的、制約などのどれとして扱うべきか早まって固定すると、モデルは最も自然な物語の中で賢く振る舞い始める。このため、`inst.md` に「入力内の要素同士の関係を早まって固定しない」という原則を追加した。
- それでも Kimi 系では初手のタスク分類が「オーディオ原因説明」に吸われ続けたため、`入力の扱い` の先頭に「回答に入る前に問いのフレーミングを確認する」内部動作を追加した。条件付きにすると、必要な場面ほど「これは該当しない」とスキップされる可能性があるため、条件は置かない。
- この種の文言は、最終回答に前提分析を露出させるためではなく、reasoning の初期経路を整えるためのものとして扱う。CoT そのものを見せる必要はないが、思考の始点を指定することで、モデルが表面上のタスク分類へ即座に吸われる確率を下げる狙いがある。
- 対照実験として、まず文言の強化はせず、同じ「問いのフレーミングを確認する」文を `入力の扱い` から instruction 先頭の `応答前の起点` へ移動した。次回の出力では、位置変更だけで CoT の初手が変わるかを見る。

**映画の好み整理プロンプト**

評価対象:

`test_prompts/05_constrained_based_curation.md` の末尾、「漠然と映画を見たい気分です...」のケース。

観測:

- いきなり推薦に行かず、まず好みの仮説を立てた点は良い。
- 「動機の明快さ」「アウトサイダーが己のルールで世界を動かす」「遊び心と過剰さへの寛容さ」という整理は、作品リストから見て筋が通っていた。
- 一方で、少ない嗜好データから「あなたは...感性を持っている」のように人物像へ踏み込みすぎる箇所があった。作品リストから言えるのは、まず「そういう作品を面白いと感じたことがある」程度であり、人格・感性への一般化は推論として扱う必要がある。

保留論点:

- v4 系にあった「推論は推論として述べる」明示を、v5 にも補強する価値があるかもしれない。現行の `認識上の透明性` はあるが、嗜好・人物像・心理傾向の推測ではやや弱く出る可能性がある。
- 末尾質問は大きな問題ではないが、誘導的・面接的になりすぎないかは継続して見る。

対応:

- `認識上の透明性` に、少数の観測や例から、相手の嗜好、性格、動機、心理傾向、能力、原因関係へ踏み込む場合は推論として扱う、という文を追加した。毎文にラベルを貼るのではなく、限られた入力から人物像や一般傾向を確定しないための補強。
