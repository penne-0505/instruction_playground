同期機能そのものを「今すぐ実装するか」と、「同期可能なデータモデルにしておくか」は分けたほうがいいです。

私なら、**同期 UI やサーバー連携は後回しにするが、データモデルだけは最初から同期を前提に設計する**、にします。

理由は、メモアプリでは後から同期を足すと、単なる通信機能では済まず、根にある前提が変わるからです。

たとえば最初にこう作ると後で苦しくなります。

```text
notes テーブル
- id: auto increment
- title
- body
- updated_at
```

これは単一端末なら自然ですが、同期を入れるとすぐに問題が出ます。

同じメモを複数端末で編集したらどちらを正とするのか。
削除済みメモをどう伝播するのか。
オフライン中に作ったメモの ID は衝突しないか。
「更新時刻が新しいほうを採用」で本文を丸ごと潰してよいのか。
将来、履歴・復元・共有を入れたくなったときに耐えるのか。

なので、実装はローカル限定でも、内部表現はこういう方向に寄せておくのが安全です。

```text
notes
- id: UUID / ULID
- created_at
- updated_at
- deleted_at nullable
- version / revision
- device_id or actor_id
```

さらに一段堅くするなら、現在状態だけでなく変更ログを持ちます。

```text
changes
- change_id
- entity_id
- actor_id
- timestamp / logical clock
- operation
- payload
```

ただし、最初から CRDT や Operational Transform を全面採用する必要はありません。自分だけのメモアプリでそこまでやると、プロダクトの主要部分より同期基盤に時間を吸われます。

私なら判断軸はこう置きます。

**同期を後付けしてよいもの**

* 同期サーバー
* アカウント機能
* 端末間ペアリング
* コンフリクト解決 UI
* E2EE
* 共有ノート
* 課金・公開向けインフラ

**最初から入れておくべきもの**

* ID は端末ローカルで一意に生成できる
* 削除は物理削除だけにしない
* 更新時刻だけに依存しない
* データ変更の境界を明確にする
* スキーママイグレーションを前提にする
* 「本文を丸ごと保存」か「編集操作を保存」かを意識して選ぶ
* ストレージ層と UI 層を密結合させない

特に重要なのは、**「同期を足す」ではなく「複数の正史を統合する」問題だと捉えること**です。
ローカルファーストでは、各端末が一時的に正しい状態を持ちます。後でそれらを合流させる。ここを最初に見ないと、同期追加時にアプリの意味論を作り直すことになります。

一方で、自分だけが当面使うなら、最初から完全な同期を作るのは過剰です。公開するか未定の段階では、同期基盤の完成度より、メモ体験・検索・編集速度・データの安心感を先に検証したほうがよいです。

したがって方針はこのあたりが現実的です。

```text
v0:
  完全ローカル
  SQLite などに保存
  UUID/ULID
  soft delete
  revision
  migration
  export/import

v1:
  手動バックアップ・ファイル同期対応
  例: Git, iCloud Drive, Dropbox folder, Syncthing などに載せても壊れにくくする

v2:
  端末間同期
  change log または revision ベースの同期
  最初は last-writer-wins でもよいが、本文衝突は検出する

v3:
  公開向け
  アカウント、E2EE、共有、競合解決 UI
```

最初の分岐は、本文の扱いです。

単純な Markdown メモなら、当初は「メモ単位で LWW + 衝突時に両方残す」で十分かもしれません。
Notion 的なブロック編集やアウトライナーを目指すなら、後から同期を足すのは重いので、ブロック単位 ID や操作ログを早めに入れたほうがよいです。

私の結論は、**同期機能は後付けでよいが、同期不能な作りにはしない**です。

ここで決めるべき問いは一つで、
**あなたのメモは「ファイル/文書」なのか、「細かい編集操作の集合」なのか**です。
前者なら後付け余地は大きいです。後者なら初期設計に同期の影を入れないと後でつらくなります。
