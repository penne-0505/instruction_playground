私なら、まず方針をこう置きます。

**Obsidianの接続は「保存時に全部やる」ものではなく、「再利用時に徐々に濃くする」ものにする。**

あなたの失敗パターンを見ると、問題はフォルダ廃止やMOC自体ではなく、**全ノートに対して最初から関係性を構築しようとしたこと**にあります。これは人間の作業負荷として続きません。リンクもタグもMOCも、全量整理の道具にすると破綻します。

## 私なら採る設計

フォルダは廃止しません。ただし、**分類ではなくライフサイクル管理**に使います。

```text
_assets/
raw/          # 外部ソース、clip、論文、記事、引用元
inbox/        # 未処理メモ、思いつき、仮置き
notes/        # 自分の考えとして残すノート
maps/         # MOC、問い、論点整理
projects/     # 期間・成果物・実務に紐づくもの
```

ここで重要なのは、`notes/` の中を細かく分けないことです。フォルダは「これは何の分野か」ではなく、「これは今どういう状態か」を表すだけにします。

ObsidianはMarkdownファイルとPropertiesを軸に動くので、構造化したい情報はfront matterに置けます。Propertiesはテキスト、リンク、日付、チェックボックス、数値、タグなどを扱える一方、Markdown本文のような複雑な表現には向かない設計です。つまり、**本文は思考、Propertiesは機械的な絞り込み**に分けるのが自然です。([Obsidian Help][1])

## タグは「内容分類」に使わない

ここはかなり重要です。

タグを `#LLM`、`#哲学`、`#設計` のような内容分類に使うと、粒度問題が必ず出ます。しかも、よく育った知識ほど複数領域にまたがるので、タグが増えるほど検索性が下がります。

私ならタグは次の用途に限定します。

```yaml
tags:
  - status/inbox
  - status/seed
  - status/evergreen
  - action/review
  - action/extract
```

つまり、タグは「このノートに何をするべきか」「成熟度はどのくらいか」を表す。内容分類はタグではなく、**リンク先の概念ノートやMOCに任せる**。

Obsidianのタグは検索やTags viewで使え、ネストタグも扱えますが、これは分類体系を巨大化させるためではなく、少数の運用状態を安定して管理する用途に向いています。([Obsidian Help][2])

## リンクは「関連」ではなく「主張」にする

`[[LLM]]`、`[[哲学]]`、`[[設計]]` のようなリンクは弱いです。つながっている理由が情報として残らないからです。

リンクを作る基準をこう変えます。

悪いリンク：

```md
これは [[LLM]] と [[知識管理]] に関係する。
```

よいリンク：

```md
この話は、[[知識管理は分類ではなく再利用のための圧縮である]] という見方に接続する。
```

つまり、リンク先はなるべく**名詞カテゴリ**ではなく、**命題・問い・論点**にする。

たとえばMOCも、

```text
LLM
哲学
Obsidian
ソフトウェア設計
```

ではなく、

```text
LLMエージェントはなぜ長期運用で破綻するのか
知識管理において分類より再発見が重要なのはなぜか
個人開発で設計判断を蓄積するにはどうすればよいか
```

のようにする。

Obsidianの内部リンクはノート同士のネットワークを作る中心機能で、ファイル名変更時に内部リンクを更新する機能もあります。なので、最初から完璧な名前にしようとせず、あとから概念ノートの名前を育てる運用と相性がよいです。([Obsidian Help][3])

## MOCは「先に作る」のではなく「同じ問いで3回戻ってきたら作る」

MOCを最初に作ると、カテゴリ表になります。それはだいたい死にます。

私ならMOC作成条件をこうします。

**同じ問い・問題意識で3回以上ノートを探したら、MOCを作る。**

例：

```md
# 知識管理において分類より再発見が重要なのはなぜか

## 現在の仮説
知識管理の失敗は、保存時の分類精度ではなく、再利用時の文脈復元にある。

## 中心ノート
- [[接続は保存時ではなく再利用時に作る]]
- [[タグは内容分類ではなく作業状態を表すべき]]
- [[MOCはカテゴリではなく問いの作業台である]]

## 関連するソース
- [[raw/Obsidian Help - Properties]]
- [[raw/Web Clipper article ...]]

## 未解決
- 自動抽出と手動リンクの境界
- LLMによる補助をどこまで信用するか
```

MOCを「目次」ではなく「思考の作業台」にする。これなら、レガシーノート全部を見直す必要がありません。

## `raw/` と `notes/` の分離は維持してよい

これはあなたの現在の運用でかなり妥当だと思います。

ただし、`raw/` を単なる墓場にしないために、rawノートには最小限のPropertiesだけ付けます。

```yaml
---
type: source
source_type: article
url: ...
author:
published:
clipped: 2026-07-05
status: unprocessed
topics:
  - "[[知識管理]]"
---
```

ただし `topics` は大量に入れない。せいぜい1〜3個です。

Web Clipperはページ内容やハイライトをObsidian vaultに保存でき、テンプレートや変数、フィルタ、ロジックで保存形式を調整できます。なので、`raw/` に入る時点で `type: source` や `status: unprocessed` を自動付与しておくと、あとで人間が分類する負荷を減らせます。([Obsidian][4])

## 新規ノートのテンプレートはこれくらいでよい

`notes/` 用：

```yaml
---
type: note
status: seed
created: {{date}}
updated: {{date}}
source:
aliases:
---
```

本文：

```md
# {{title}}

## 要点
-

## 自分の主張
-

## 接続
- 上位の問い:
- 反対・緊張関係:
- 具体例:
```

ここで「接続」を強制しすぎないほうがよいです。新規作成時に必須にするのは、せいぜい1つだけ。

特に有効なのは、リンクを3種類に分けることです。

```md
上位の問い: [[...]]
反対・緊張関係: [[...]]
具体例: [[...]]
```

単なる「関連」よりも、関係の型が残るので後で読み返したときに意味があります。

## Basesを「整理画面」として使う

今なら、`status: unprocessed` や `type: source` を持つノートを一覧する用途にBasesを使うのがよさそうです。BasesはObsidianのコアプラグインで、MarkdownファイルとPropertiesをもとに、ノートをテーブル、リスト、カード、地図などのビューで表示・編集・フィルタできます。([Obsidian Help][5])

使い方としては、こうです。

```text
maps/_review.base
```

に以下のようなビューを作る。

* 未処理ソース一覧
* `status: seed` のノート一覧
* `action/review` タグ付きノート一覧
* 最近30日で更新したノート
* リンクが少ないが何度も開いているノート

ここでやるべきことは、「きれいなデータベースを作る」ではありません。**次に手を入れるべきノートを見つける**ことです。

## レガシーノートは移行しない

これは明確にそうしたほうがよいと思います。

レガシー全体を整理対象にすると、Obsidian運用が「過去の負債返済」になります。そうではなく、次のルールにします。

**触ったノートだけ、30秒以内で最低限直す。**

たとえば既存ノートを開いたら、次のどれか1つだけやる。

```text
- type/status だけ付ける
- 近いMOCに1本だけリンクする
- タイトルを命題っぽく変える
- source を1つだけ足す
- 逆に、何もしないで閉じる
```

この「何もしないで閉じる」を許可しないと、Obsidianを開く心理的コストが上がります。

## 私の暫定結論

あなたの場合、目指すべきはZettelkasten風の完全リンク網ではなく、次のような構造だと思います。

```text
raw/      = 証拠・素材
notes/    = 自分の命題
maps/     = 問いごとの作業台
projects/ = 期限や成果物のある活動
tags      = 作業状態
properties = 機械的フィルタ
links     = 思考上の主張関係
```

一番大きな変更点は、**「有機的につなぐ」を分類問題として扱わないこと**です。

有機的な接続は、ノートを保存するときではなく、

```text
読み返した
使った
反論した
別の文脈に転用した
MOCに載せた
```

ときに発生する。

なので、運用の中心は「保存」ではなく「再訪」です。Obsidianで失敗しやすいのは、入口を整えすぎて、出口、つまり再利用の導線がないことです。

まず試すなら、次の3ルールだけでよいと思います。

```text
1. タグは status/action 以外に増やさない
2. MOCはカテゴリ名ではなく問いの形で作る
3. 既存ノートは触ったものだけ30秒以内で改善する
```

この運用だと、`raw/` はそのまま活かせます。`notes/` は徐々に命題ノート化します。`maps/` は必要になった問いだけが育ちます。
最初に詰めるべき分岐は、あなたのObsidianが主に「研究・思考の蓄積」なのか、「個人開発や実務判断の再利用」なのかです。ここでMOCの粒度が変わります。
