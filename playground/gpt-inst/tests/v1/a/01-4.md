最初にやるべきことは、**詳細な要件定義**というより、**「この画面をどの場面で開くのか」を1つに決めること**だと思います。

参照リポジトリは、README上では「Google検索画面や新しいタブをGoogle Nest Hub風のスマートディスプレイに変える」拡張機能で、天気・ニュース・時計・メディア連携・Zenモード・焼き付き防止・細かい見た目設定を持っています。([GitHub][1]) 実装上も、Chrome拡張として新規タブを上書きし、`storage`、`tabs`、`scripting`、`history`、`search` などの権限を使う設計です。 つまり、これは「普通のWebページ」ではなく、ブラウザの新規タブ・履歴・検索・メディア状態に近い場所へ入り込む設計です。

あなたの場合はChrome拡張に限定しないとのことなので、まずこう切り分けるのがよいです。

## 1. まず決めるべき問い

「自分専用ダッシュボード」は、機能名ではなく**利用場面**から決めたほうが失敗しにくいです。

たとえば、どれを作るのかで設計が変わります。

* **新規タブ型**: 作業開始時に見る。検索、リンク、今日の予定、メモが中心。
* **常時表示型**: サブディスプレイやタブレットに置く。時計、天気、予定、音楽、通知が中心。
* **学習管理型**: 毎日見る教材・読書・実装ログ・復習カードが中心。
* **個人OS型**: タスク、メモ、RSS、カレンダー、GitHub、家計、習慣などを統合する。

最初から全部を混ぜると、「美しいけど使わない画面」になりやすいです。
最初の要件は、こういう1文で十分です。

> 朝と作業開始時に開き、今日やること・予定・学習対象・よく使うリンクを10秒で確認できるセルフホストWebダッシュボード。

この1文が決まると、必要な機能と不要な機能が分かれます。

## 2. Chrome拡張にしないなら、最初の技術選択

私は最初は **セルフホストWebアプリ**でよいと思います。理由は、学習対象として扱いやすいからです。

Chrome拡張にすると、`manifest.json`、権限、content script、background service worker、ブラウザAPI、CSP、OAuthリダイレクトなど、学習できることは多い一方で、ダッシュボード本体より「拡張機能特有の都合」に時間を取られます。参照リポジトリも、Spotify認証、YouTube/Spotify連携、Open-Meteo取得、履歴検索などをbackground側で扱っています。

最初の構成案はこれで十分です。

```txt
dashboard/
  app/ or src/
    components/
    modules/
      Clock.tsx
      Weather.tsx
      Calendar.tsx
      Links.tsx
      LearningQueue.tsx
      Memo.tsx
    lib/
      db.ts
      rss.ts
      weather.ts
  data/
    dashboard.sqlite
  docker-compose.yml
```

技術スタックは、学習目的なら次のどちらかが扱いやすいです。

**堅実案**: Next.js / SvelteKit + TypeScript + SQLite
**軽量案**: Vite + React/Svelte + Hono/Express + SQLite

個人開発で「UIもAPIもDBもセルフホストも触りたい」なら、私は **SvelteKit or Next.js + SQLite + Docker** を推します。Reactを深めたいならNext.js、状態とUIの見通しを重視するならSvelteKitです。

## 3. 最初のMVP

最初のMVPは、機能を少なくして「毎日開く理由」を作るべきです。

私なら最初はこの5つにします。

1. **時計・日付**
2. **今日の予定**
3. **今日のタスク / メモ**
4. **学習キュー**
5. **クイックリンク / 検索**

天気・ニュース・Spotify・RSS・GitHub連携などは魅力的ですが、最初から入れると外部API処理が主役になります。参照リポジトリにも天気、ニュース、カウントダウン、TODO、電卓、地震情報、為替などのモジュール定義があります。 ただ、あなたの目的が「自分専用 + スキル向上」なら、まずは**自分の行動ログや学習対象を扱うモジュール**を中心に置いたほうがよいです。

たとえば、学習キューはこういう形にできます。

```txt
今日の学習
- 読む: TypeScript 5.x release note
- 作る: dashboard の Weather module
- 試す: SQLite migration
- 書く: 実装メモ 5行
```

ここに「昨日やったこと」「途中の実装課題」「次に読むドキュメント」を出すと、単なる表示画面ではなく、学習の継続装置になります。

## 4. 要件定義はこの粒度で始める

最初に書く要件は、詳細な仕様書ではなく、以下の表くらいでよいです。

| 項目     | 仮決め                    |
| ------ | ---------------------- |
| 使う場面   | 作業開始時・朝・休憩明け           |
| 表示端末   | PCブラウザ、将来的にタブレット       |
| 更新頻度   | 時計はリアルタイム、予定は数分〜数十分単位  |
| 保存するもの | タスク、メモ、リンク、学習ログ、設定     |
| 外部連携   | 最初は天気とカレンダー程度          |
| 認証     | 最初はローカルネットワーク前提、後で認証追加 |
| 最重要体験  | 開いた瞬間に「今日何をすればよいか」が分かる |

ここで重要なのは、**「何を表示するか」より「何を判断できるようにするか」**です。

たとえば、

```txt
悪い要件:
天気、ニュース、時計、TODO、カレンダー、RSSを表示する。

よい要件:
作業開始時に、今日の予定・現在の集中対象・次に進める学習タスクを10秒以内に確認できる。
```

後者なら、UIもデータ設計も判断しやすくなります。

## 5. 実装の進め方

順番はこれがよいです。

### Phase 1: 静的ダッシュボード

DBもAPIもなしで、まず画面を作ります。

* レイアウト
* 時計
* 固定リンク
* 固定メモ
* ダミーの予定
* ダミーの学習キュー

ここで「毎日見たい見た目か」を確認します。
この段階でデザインに少し時間を使うのは有益です。ダッシュボードは使われないと意味がないので、見たくなる画面であることは機能要件の一部です。

### Phase 2: ローカル永続化

次にSQLiteを入れます。

* links
* todos
* notes
* learning_items
* settings

最初のDBはこれくらいで足ります。

```sql
learning_items (
  id,
  title,
  kind,        -- read / build / watch / write
  source_url,
  status,      -- inbox / today / done / archived
  priority,
  created_at,
  completed_at
)
```

### Phase 3: 外部データ取得

この順で入れるのがよいです。

1. 天気
2. カレンダー
3. RSS
4. GitHub
5. Spotify / YouTube

SpotifyやYouTubeの現在再生情報は楽しいですが、OAuthや各サービスの制約があり、初期実装の重心がずれます。参照リポジトリでもSpotifyのPKCE認証やトークン更新、再生状態取得をbackground側で処理していて、地味に重い領域です。

### Phase 4: 自分用の編集体験

ここが一番重要です。

ダッシュボードは「表示する場所」ではなく、「日々更新される自分用データの入口」にすると価値が出ます。

* 1キーでメモ追加
* 今日やる項目を上に送る
* 学習項目を完了にする
* リンクをカテゴリ分けする
* 使っていないモジュールを非表示にする

参照リポジトリもカスタマイズ性を特徴としていて、背景の明るさ、ガラスの透明度、時計フォントなどを調整できると説明されています。([GitHub][1]) ただし、自作では見た目設定より先に、**自分の行動データを更新しやすいこと**を優先したほうが使い続けやすいです。

## 6. 最初の完成形のイメージ

最初のバージョンはこのくらいがよいです。

```txt
┌──────────────────────────────┐
│  09:42   2026-07-05 Sun       │
│  今日の焦点: Dashboard MVP     │
├──────────────┬───────────────┤
│ 今日の予定     │ 学習キュー       │
│ 10:00 ...     │ Build: Weather │
│ 14:00 ...     │ Read: OAuth    │
├──────────────┼───────────────┤
│ TODO          │ Links / Search │
│ □ ...         │ GitHub Docs    │
│ □ ...         │ Localhost      │
└──────────────┴───────────────┘
```

ここに後から、天気、RSS、GitHub activity、読書ログ、LLMプロンプト集、Zenn/Qiita下書き、Anki風復習などを足せます。

## 7. 学習目標として分解する

このプロジェクトは、機能単位よりも学習単位で切るとよいです。

* UI設計: レスポンシブ、カードレイアウト、キーボード操作
* 状態管理: local state / server state / optimistic update
* DB設計: SQLite、migration、軽い正規化
* API設計: moduleごとのfetcher、キャッシュ、失敗時表示
* セルフホスト: Docker、環境変数、バックアップ
* 認証: 自宅LAN限定からOAuth/Basic認証へ
* 運用: cron、ログ、データ移行、設定エクスポート

この順に進めると、見た目だけの習作ではなく、小さい個人SaaSのような練習になります。

## 私なら最初にやること

最初の1コミットは、機能ではなくこれです。

```txt
docs/vision.md
```

中身は5行でよいです。

```md
# Personal Dashboard

## Purpose
作業開始時に、今日の予定・集中対象・学習キューを10秒で確認する。

## First Surface
PCブラウザで開くセルフホストWebページ。

## MVP Modules
Clock, Today, Tasks, Learning Queue, Quick Links.

## Non-goals
Chrome拡張、Spotify連携、ニュース集約、凝った設定画面は初期対象外。

## Success
1週間、自分が毎日1回以上開いたら成功。
```

その次に、ダミーデータでUIを作る。
要件を詰めるのは正しいですが、最初から完全な要件定義をすると手が止まりやすいです。**「使う場面」と「MVPモジュール」だけ決めて、画面を先に出す**のがよいと思います。

最初の分岐としては、あなたのダッシュボードを **作業開始用** にするか、**常時表示用** にするかです。ここだけは先に決めたほうが、UI密度・更新頻度・入力機能の設計がかなり変わります。
