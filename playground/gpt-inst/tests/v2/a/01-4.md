`https://github.com/naikaku1/Search-Immersion`

このリポジトリのような私専用のダッシュボードを、スキル向上の意図も含めて実装したいと考えています。何から始めるべきでしょうか？私はまずは要件をはっきりさせるべきかなと考えています。ただし、Chrome拡張で実装する必要はなく、セルフホストの普通のwebページのようなものでも構いません。要旨は"自分専用のダッシュボードを学習を兼ねて作成したい"です。

















参照リポジトリは「個人用ダッシュボード」というより、**Chrome の新規タブ／Google検索画面をスマートディスプレイ風に置き換える拡張**ですね。README上の主機能は、時計・天気・ニュース・Zenモード・メディア連携・設定画面などで、実装も Manifest V3 の Chrome拡張として `newtab.html` を上書きし、`storage` や `history`、YouTube/Spotify/Open-Meteo などへの権限を持つ構成です。([GitHub][1])

なので、最初にやるべきことは「要件定義」ですが、**普通の業務アプリ的な要件定義ではなく、日常使用と学習対象を分けること**だと思います。

まずこう切るのがよいです。

## 1. 最初に決めるのは「何を表示するか」ではなく「いつ見るか」

ダッシュボードは表示項目を増やすほど作りやすく見えて、使われなくなりがちです。最初はこのどれかに絞るべきです。

1. **朝に開くページ**
   今日の予定、天気、タスク、習慣、ニュース少量。

2. **作業開始時に開くページ**
   今日やること、最近触っているリポジトリ、メモ、リンク、タイマー。

3. **常時表示するページ**
   時計、予定、音楽、通知、集中モード、焼き付き対策。

Search Immersion は 3 に近いです。常時表示・新規タブ・メディア連携・視覚演出が主軸です。([GitHub][1])
一方、セルフホストのWebページで始めるなら、最初は **1 or 2** のほうが実装量と学習効率のバランスがよいです。

## 2. 最初のMVPは「3カード + 設定保存」くらいでよい

私なら最初の到達点をこうします。

```text
自分専用ダッシュボード v0

目的:
  作業開始時に開き、今日の状態を30秒で把握する。

画面:
  - 現在時刻・日付
  - 今日の予定または今日やること
  - よく使うリンク / 最近のメモ / 天気 のどれか1つ

機能:
  - カードの表示/非表示を設定できる
  - 設定を localStorage または SQLite に保存する
  - レスポンシブ対応
  - セルフホストでLAN内または自分だけが見られる
```

ここで重要なのは、**天気・ニュース・Spotify・カレンダー・検索・壁紙・認証を全部同時にやらない**ことです。
最初から外部APIを増やすと、UI設計ではなく認証・CORS・トークン管理・API仕様変更対応が主作業になります。

## 3. 技術選定は「学びたい層」で決める

Chrome拡張を学びたいわけでないなら、最初は普通のWebアプリでよいです。拡張にすると、Manifest、permissions、service worker、content script など別の学習対象が入ってきます。Search Immersion もまさにその構成です。([GitHub][2])

選択肢は大きく3つです。

### A. UI実装を学びたい

**Vite + React / Svelte / Vue**

Vite は現代的なフロントエンド用ビルドツールとして公式に案内されています。まず静的なSPAで作るなら軽いです。([vitejs][3])

向いているもの:
時計、カードUI、設定パネル、テーマ切替、localStorage、アニメーション。

### B. フルスタックを学びたい

**SvelteKit / Next.js**

SvelteKit はWebアプリケーションを作るためのSvelteベースのフレームワーク、Next.js はReactベースでApp RouterやServer Componentsを扱えるフレームワークです。([Svelte][4])

向いているもの:
SQLite、認証、RSS取得、カレンダー連携、GitHub API、サーバー側キャッシュ。

### C. ブラウザ統合を学びたい

**Chrome拡張**

向いているもの:
新規タブ置換、閲覧履歴、現在再生中メディア、ブラウザ操作との統合。

ただしこれは第2段階でよいです。最初から拡張にすると、ダッシュボード本体の設計が進む前に権限周りで足を取られます。

## 4. 私ならこの順で始めます

最初の一歩は、READMEや画面モックではなく、**「毎日見る1画面の仕様メモ」**です。

```text
このページを開く場面:
  作業開始時

最初に見たい情報:
  1. 今日やること
  2. 今日の予定
  3. よく使うリンク

操作:
  - タスクを追加/完了できる
  - リンクを編集できる
  - カードの並び替えは後回し

保存:
  v0 は localStorage
  v1 で SQLite

外部API:
  v0 は使わない
  v1 で天気 or カレンダーを1つだけ追加

学習テーマ:
  - コンポーネント設計
  - 状態管理
  - 永続化
  - デプロイ
```

これを書いたら、すぐ実装に入ってよいです。要件を詰めすぎるより、**毎日開いて違和感を拾う**ほうがこの手の個人ツールでは精度が上がります。

## 5. 最初の実装スコープ案

ぺんねさんの意図なら、私はこう始めるのを勧めます。

```text
v0: ローカル完結
- Vite で開始
- 1ページ構成
- ClockCard
- TodoCard
- LinkDock
- SettingsDrawer
- localStorage 永続化

v1: セルフホスト化
- Docker化
- 設定ファイルまたはSQLite
- LAN内アクセス
- バックアップ/インポート

v2: 外部データ
- Open-Meteo 天気
- RSS
- Google Calendar もしくは iCal URL
- GitHub activity

v3: ブラウザ統合
- 必要なら Chrome拡張化
- 新規タブ置換
- Web版と共通UIを使う
```

Chrome拡張にするかどうかは、最後でよいです。
**「新規タブに出したい」だけなら拡張化の価値がありますが、「自分用の情報面を育てたい」ならセルフホストWebから始めるほうが筋がよい**です。

最初に決めるべき1点は、**これは朝の確認ページですか、作業開始ページですか、常時表示ページですか。**
