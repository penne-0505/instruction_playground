正直、あなたの失敗パターンはかなり典型的だと思います。
Obsidianで「知識を有機的に繋ぐぞ」と意気込むと、だいたい **“全ノートを正しく分類・リンクしなければならない”** という地獄に落ちます。これはもう知識管理というより、知識の確定申告です。

私なら、次の方針にします。

## 結論：Obsidianは「分類システム」ではなく「発見システム」として運用する

あなたの場合、ベストプラクティスはたぶんこれです。

> **フォルダは保管場所、タグは状態、Propertiesは検索・一覧、リンクは思考の接続、MOCは入口。**

これを混ぜると破綻します。
以前の運用では、タグ・front-matter・リンク・MOCが全部「分類」を担おうとしていたのが苦しかったのだと思います。

---

## 1. フォルダは廃止しない。ただし「意味分類」には使わない

今の構成は悪くないです。

```text
_assets/
raw/
notes/
```

むしろ良いです。
私なら少しだけ増やして、こうします。

```text
_assets/
_templates/
raw/
notes/
maps/
daily/
```

役割はこうです。

```text
raw/    = 外部から取ってきた素材。Web記事、PDFメモ、引用、一次情報。
notes/  = 自分の言葉で書いた知識・考察・仮説。
maps/   = MOC、テーマ別入口、プロジェクト別入口。
daily/  = その日の作業ログ、雑メモ、思考の一時置き場。
_assets/ = 画像、添付、テンプレート素材。
```

大事なのは、フォルダを「心理学」「仕事」「技術」「読書」みたいな意味分類にしないことです。
意味分類をフォルダに背負わせると、1つのノートが複数領域にまたがった瞬間に詰みます。

---

## 2. タグは「内容」ではなく「状態」にだけ使う

タグの粒度で迷ったとのことですが、これはかなり自然です。
内容タグは増殖します。

たとえばこういうタグは破綻しやすいです。

```text
#AI
#LLM
#哲学
#仕事術
#教育
#認知科学
#読書
#メモ術
```

なぜなら、これは分類体系をタグで作ろうとしているからです。

代わりに、タグはワークフローに限定します。

```text
#inbox
#processing
#evergreen
#review
#draft
#project
#source
#moc
```

つまり、タグは「これは何についてのノートか」ではなく、**「今どう扱うべきノートか」** を示すものにします。

Obsidian公式でも、タグはノートを見つけるためのキーワードやトピックとして説明されていますが、ネストタグも含めて検索・フィルタ用の仕組みです。だからこそ、使いすぎると第二のフォルダ地獄になりがちです。([Obsidian][1])

私なら内容分類はタグではなく、後述する `topics` やリンクに寄せます。

---

## 3. Propertiesは「機械に読ませる情報」だけにする

front-matterはかなり有効です。
ただし、ここも欲張ると死にます。

ObsidianのPropertiesは、テキスト、リンク、日付、チェックボックス、数値、タグなどの構造化データを扱えます。さらにBasesなどと組み合わせると、ノートを表・カード・リストのように表示・絞り込みできます。([Obsidian][2])

私なら、まずこのくらいに絞ります。

```yaml
---
type: note
status: seed
created: 2026-07-05
updated: 2026-07-05
topics:
  - "[[知識管理]]"
  - "[[Obsidian]]"
source:
related:
---
```

`raw/` のWebクリップならこう。

```yaml
---
type: source
source_type: article
status: unread
title: 
author:
url:
published:
clipped: 2026-07-05
topics:
  - "[[Obsidian]]"
---
```

ポイントは、**topicをタグではなくリンクにする** ことです。

```yaml
topics:
  - "[[知識管理]]"
  - "[[ノート術]]"
```

こうすると、`[[知識管理]]` 自体をMOCにもできます。
タグよりも「そこに説明文や問いを書ける」のが強いです。

---

## 4. リンクは「分類」ではなく「文章中の意味接続」に限定する

リンクが機能しなかった理由は、たぶん「後から既存ノートを全部リンクしようとした」からです。
これは無理です。人類には早すぎる。

リンクはこういうときだけ張るのがいいです。

```md
この話は [[知識管理は検索ではなく再発見の仕組みである]] という考えに近い。
```

または、

```md
この論文の主張は [[人間は分類より文脈で記憶する]] の具体例として読める。
```

つまりリンクは、ノート同士を「関連しているから」繋ぐのではなく、**自分の文章の中で必要になったときだけ繋ぐ**。

リンク数の目安は、1ノートあたり最初は **1〜3個** で十分です。
ゼロでもいいです。

ObsidianにはBacklinks機能があり、リンク済み言及だけでなく、ノート名が本文に出ているがリンク化されていない「Unlinked mentions」も確認できます。なので、最初から完璧にリンクしなくても、後から発見できます。([Obsidian][3])

ここはかなり重要です。

> **リンクは作業ではなく、発見されたときに張る。**

---

## 5. MOCは「索引」ではなく「編集された入口」にする

MOCを全体の目次にしようとすると、更新作業が重くなります。
なので、MOCは「完璧な一覧」ではなく、**そのテーマについて考えるための入口ページ** にするのがよさそうです。

たとえば `maps/知識管理.md` はこう。

```md
---
type: moc
status: active
topics:
  - "[[知識管理]]"
---

# 知識管理

## 今の中心的な問い

- 情報を保存することと、あとで使える知識にすることは何が違うのか？
- Obsidianでは分類よりも再発見の導線が重要ではないか？

## 重要ノート

- [[知識管理は検索ではなく再発見の仕組みである]]
- [[タグは内容分類より状態管理に使う]]
- [[MOCは索引ではなく思考の入口である]]

## 関連する素材

- [[2026-07-05 Obsidian運用について考える]]
- [[source - Obsidian Web Clipper article xxx]]

## 未解決

- rawの蓄積が増えたとき、どこまで加工するべきか？
- project noteとevergreen noteは分けるべきか？
```

MOCは「全ノートの網羅表」ではなく、**自分がそのテーマに戻ってきたときに、すぐ思考を再開できるページ** です。

---

## 6. Basesを「自動MOC」として使う

ここは今のObsidianならかなり使えます。
Basesは公式のCore pluginで、ノートのPropertiesをもとに、表示・編集・ソート・フィルタできるデータベース的ビューを作れます。ビューは表、リスト、カードなどで表示できます。([Obsidian][4])

つまり、MOCに手で全部リンクを貼る必要はありません。

たとえば、こういうビューを作る。

* `status = unread` の source 一覧
* `type = source` かつ `topics contains [[Obsidian]]`
* `type = note` かつ `status = seed`
* `status = evergreen`
* `updated が古い active note`

手でMOCを整備するのではなく、**Propertiesでざっくり絞り込み、MOCには本当に重要な数件だけ手で置く**。

これがかなり現実的です。

---

## 7. rawとnotesの分離は正しい。ただし「加工ルール」を決める

あなたの「ソースと考えの分離」はかなり良いです。
これは続けるべきです。

ただ、`raw/` が増えすぎると、ただの資料墓場になります。
なので、Webクリップには最低限の加工ルールを設けます。

Web Clipperは公式にカスタムテンプレートや変数、条件分岐、ループを使える仕組みがあり、クリップ内容をローカルのVaultに保存できます。([Obsidian][5])

`raw/` の各ファイルには、本文の上にこの3項目だけ足すといいです。

```md
## Why clipped?

- 

## Useful parts

- 

## My take

- 
```

全部読まなくてもいいです。
ただし、保存した瞬間に1行だけでも「なぜ取ったか」を書く。

これをしないと、未来の自分が見ても「で？」になります。未来の自分、意外と他人です。

---

## 8. レガシーノートは移行しない。「触ったときだけ整える」

過去ノートを全部見直すのはやめたほうがいいです。
それは整理ではなく、沼掃除です。

ルールはこれで十分です。

> **古いノートは、開いたときだけ30秒メンテする。**

30秒でやることは3つだけ。

```text
1. type と status を入れる
2. topics を1〜3個入れる
3. 本文中で自然なリンクを1つだけ貼る
```

例：

```yaml
---
type: note
status: seed
topics:
  - "[[Obsidian]]"
  - "[[知識管理]]"
---
```

これだけでいいです。
全Vaultを一気に整理しようとしない。**よく使うノートだけが徐々に豊かになる** 形にします。

これはむしろ自然です。
使わないノートを完璧に整備する意味は薄いです。

---

## 9. ノートの種類を明確にする

あなたのVaultでは、たぶん最低限この5種類で足ります。

```text
source   = 外部資料。rawに置く。
note     = 自分の考え。notesに置く。
moc      = テーマ入口。mapsに置く。
daily    = 一時ログ。dailyに置く。
project  = 進行中の作業や成果物。
```

Propertiesではこう。

```yaml
type: source
```

または、

```yaml
type: note
```

タグで `#source` とするより、`type` propertyにしたほうが一覧・フィルタ・Basesで扱いやすいです。

---

## 10. おすすめの実運用フロー

### A. Web記事を見つけたとき

`raw/` に保存。

```yaml
---
type: source
status: unread
source_type: article
url:
clipped:
topics:
  - "[[Obsidian]]"
---
```

本文の冒頭に：

```md
## Why clipped?

- ObsidianのMOC運用について参考になりそうだったため。

## Useful parts

- 

## My take

- 
```

この時点ではリンクしすぎない。

---

### B. 読んだとき

`status` を変える。

```yaml
status: processed
```

そして `notes/` に自分のノートを1つだけ作る。

```md
# MOCは索引ではなく思考の入口である

MOCを全ノートの目次として作ると更新コストが高すぎる。
むしろMOCは、そのテーマについて再び考え始めるための入口として使うべきだ。

## Source

- [[rawの元記事]]

## Related

- [[知識管理]]
- [[タグは状態管理に限定する]]
```

---

### C. MOCに入れるか判断する

MOCに入れる条件はこれくらいでいいです。

```text
- そのテーマについて3回以上考えた
- 関連ノートが5個以上になった
- 自分が後で戻ってきそう
- プロジェクトや文章に使う可能性がある
```

逆に、ノートが1〜2個しかないテーマでMOCを作る必要はないです。

---

## 私なら採用するルール

かなり圧縮すると、こうです。

```text
1. フォルダは「保管場所」だけに使う。
2. タグは「状態」だけに使う。
3. Propertiesは一覧・検索・Bases用に使う。
4. topicはタグではなく [[リンク]] にする。
5. rawは外部資料、notesは自分の考えとして分ける。
6. MOCは手動索引ではなく、思考の入口にする。
7. レガシーノートは一括整理しない。触ったときだけ直す。
8. 1ノートに最初から大量リンクを張らない。1〜3個でよい。
9. Webクリップには必ず「なぜ保存したか」を1行書く。
10. Basesで“未処理・種・成熟済み”を一覧化する。
```

---

## たぶん一番効く変更

あなたの場合、最も効くのはこれだと思います。

> **「分類したい」という欲求を、タグではなく `type / status / topics` に分解する。**

つまり、

```yaml
type: note
status: seed
topics:
  - "[[知識管理]]
  - "[[Obsidian]]
```

この3つで十分に管理できます。

分類を1つの仕組みに背負わせない。
フォルダ、タグ、Properties、リンク、MOCにそれぞれ別の仕事をさせる。

Obsidianの良さは「全部がリンクすること」ではなく、**必要になったときに関係が浮かび上がること** だと思います。
なので、最初から有機的なネットワークを作ろうとせず、日々の使用によって自然に濃くなる設計にしたほうが合います。
